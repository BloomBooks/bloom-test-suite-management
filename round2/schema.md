# Round2 Import Schema

## Purpose

This document is the round2 handoff reference.

It is written for someone starting cold who needs to understand:

- what the round2 JSON files are
- how they are produced from the CSV
- how those fields map into the single live Notion database
- which parts of the workflow are preparation versus transport

If this file and the code disagree, the code is authoritative. In practice, the round2 pipeline is intentionally split so that:

- `prepare-import.mjs` interprets and normalizes spreadsheet content
- `import-to-notion.mjs` mostly transports prepared values into Notion

## Model Change (current plan)

After testing, the model was simplified to a **single Notion database: `Test Case Runs`.**

- There is **no** separate `Test Cases` database. Durable case metadata is folded onto every run card as its own properties.
- There is **no** separate `Test Suite Runs` database. Suite-run membership is recorded as a closed `Test Suite Run` **select tag** on each run card, not as a relation.

The old `Test Cases` and `Test Suite Runs` databases that may still exist in the workspace are intentionally left untouched. The importer simply stops writing to them, and they are dropped from `notion-config.json`.

### Clean slate on the Notion side

The import now starts from a clean slate under a new root page, `Bloom-Tests` (`37d4bb19df128097a7f9f7f0ab9f1a2f`). `import-to-notion.mjs` creates a fresh `Test Case Runs` database inside that page on first run:

- If `notion-config.json` has a non-empty `databases.testCaseRuns`, that id is used.
- Otherwise, if `notion-state.json` already records a created `databaseId`, that one is reused.
- Otherwise the importer creates a new `Test Case Runs` database under `parentPageId`, writes its id into `notion-state.json` as `databaseId`, and prints it.

A freshly created database already has the full property schema, so live-schema reconciliation is skipped on the creating run.

On creation only, the importer also hides the plumbing properties on the default
table view via the **Views API** (`hideDefaultViewProperties`, which needs API
version `2025-09-03`): it lists the database's views, finds the default one, and
sets `visible: false` for `HIDDEN_VIEW_PROPERTIES` (`Import Run ID`,
`Source Row Number`, `Test Case ID`, `Legacy Number`, `Dokimion ID`). This is
best-effort — a failure is reported in the run output (`viewVisibility`) but does
not abort the import. It runs only when the database is newly created, so a view
the user later customizes is never overwritten. (Select-option colors for
`Priority` and `Status` are likewise applied at creation; see their mappings.)

The same properties cannot be hidden on the **opened-card** view via the API —
that is a manual UI step listed in `manual-steps-after-notion-import.md`.

## Imported Version Range

Only suite runs from `6.4` down to `4.8` are imported. `detectSlots()` stops
when it reaches the `4.7 FX` top-header (`STOP_AT_TOP_HEADER`); `4.7 FX` and
everything older is excluded. Those older columns use an irregular platform-split
layout (`Person testing <ver> <platform>`, with Windows groups carrying a shared
"any platform" issues column and Wasta groups one column shorter) that the slot
detector mis-aligned, dropping data. Rather than parse it, the import is cut off
before it. As a result every run card maps to exactly one execution.

## Files And Roles

### Inputs

- `Bloom Test Plan.csv`
  The source spreadsheet.

- `round2/notion-config.json`
  Stores `parentPageId` (the `Bloom-Tests` root page) and `databases.testCaseRuns`. Leave `testCaseRuns` empty (`""`) to have the importer create a fresh database under `parentPageId`.

- `round2/area-mapping.json`
  Maps section-header rows to an area / instruction / ignore (see Area).

- `round2/case-summaries.json`
  Hand-written `snapshot -> summary` titles (see Case Summaries).

### Generated By `prepare-import.mjs`

The default output folder is `round2/build/`.

- `test-case-runs.json`
  The only Notion-bound file. One record per `(case, suite run)` pair that has execution data, with durable case metadata folded in.

- `suite-run-tags.json`
  The distinct, ordered list of `Test Suite Run` tag values discovered in the spreadsheet. This documents the closed select list; the importer also creates options on demand.

- `prepare-summary.json`
  Counts and metadata about the preparation pass.

- `date-warnings.json`
  Rows whose run-date values could not be normalized cleanly.

### Generated By `import-to-notion.mjs`

- `notion-state.json`
  Import state keyed by `importRunId`. This is how the importer knows whether to create or update a run card.

- `notion-import-failures.json`
  Append-only failure log for case runs.

The importer also writes body content into Notion run pages and can reconcile the live `Test Case Runs` schema.

## End-To-End Flow

1. Parse the spreadsheet.
2. Detect run-column slots (one per suite run / platform).
3. For each importable row, derive one logical test case (kept in memory only).
4. Derive one `Test Case Run` per `(case, suite run)` pair that has execution data, folding the case metadata onto it.
5. Collect the distinct suite-run names as the closed `Test Suite Run` tag list.
6. Upload the prepared run records into the single Notion database.

The main design rule is:

- semantic interpretation happens in preparation
- Notion upload should not need to rediscover meaning from raw prose

## Core Entity

The round2 model now has one persisted entity.

### `Test Case Run`

One instance of a test case in one suite run. This is the actionable work card testers use in Notion, and it is the only Notion database.

Each run card carries:

- the run-specific execution data (assignee, date, build, issues, status)
- the suite-run identity as a `Test Suite Run` select tag
- the durable case metadata (title summary, legacy number, dokimion id, priority, past issues, estimated time) as its own properties
- the test description / steps in the page body

The "test case" and "test suite run" still exist as concepts during preparation, but they are not separate databases. A test case is simply the set of run cards that share the same `Test Case ID`; a suite run is simply the set of run cards that share the same `Test Suite Run` tag.

## `test-case-runs.json`

Each object represents one actionable run card.

### Identity

- `importRunId`
  Stable primary key for the run card. Format: `<case import slug>::suiteRunKey` (the case slug comes from `buildImportId`; it is no longer a record field of its own). Used as the import-state key and for dedup, and imported into the `Import Run ID` property.

- `suiteRunKey`
  Slug form of the suite-run name. Used only to build `importRunId`; not sent to Notion.

- `testCaseId`
  Numeric test case ID, identical for every suite run of the same case. Currently the source CSV row number (unique per logical case). Imported into the `Test Case ID` number property — the way to group all run cards of one case.

- `sourceRowNumber`
  Original CSV row number.

### Suite-run tag

- `suiteRunTag`
  Human-readable suite-run name (e.g. `6.3`, `4.9 Spot Testing`). Imported into the `Test Suite Run` select property. This is the closed selectable list that replaces the old suite-run relation. Names are normalized in `prepare-import.mjs` by `normalizeSuiteRunName()`, which strips the `BetaInternal` qualifier (e.g. `5.4 BetaInternal` -> `5.4`); the slugified key is recomputed from the normalized name.

### Folded case metadata

- `caseSummary`
  A short (≤5–6 word) **hand-written** title, looked up from `case-summaries.json` by the snapshot text. Used as the **card title** (`Test Case Run`); not stored in any other property. Blank for snapshots not yet authored (see Case Summaries).

- `area`
  The functional area this test belongs to (see Area), carried forward from the most recent section-header row. Imported into the `Area` select.

- `legacyNumber`
  Spreadsheet legacy identifier when present.

- `dokimionId`
  External test identifier.

- `priority`
  Normalized priority label. Allowed values normalize to `1`, `2`, `3`, `Ignore`, or `Duplicate`.

- `pastIssues`
  Historical issue references, often `BL-` IDs. Rendered with hyperlinks.

- `estTimeMin`
  Estimated time in minutes, or `null`.

- `caseSnapshot`
  The cleaned case description with line breaks preserved (each line trimmed, leading bullet dash removed, internal whitespace collapsed, empty lines dropped). Rendered into the run-card page body under "Test Case Snapshot", and the lookup key for `caseSummary`.

### Execution fields

- `assignees`
  Array of canonical tester names, normalized from the Person column (see Assignee Normalization). Imported into the `Assignee` multi-select. Empty on Skipped runs.

- `notes`
  Free-text notes. Sourced from a Person value that isn't a clean name or skip (see Assignee Normalization) and/or leftover date text that didn't parse (see Date Handling). Multiple notes are joined with `; `. Imported into the `Notes` rich_text.

- `testedOn`
  Normalized ISO-like date string when available.

- `buildTested`
  Build information from the execution column.

- `issueLinks`
  Issue references from the execution column. Rendered with hyperlinks.

- `status`
  Execution status (`Skipped` / `Done` / `""`); see Status Derivation. Imported into the `Status` select. There is no card-level `ok` field and no OK checkbox.

## `suite-run-tags.json`

Each object documents one `Test Suite Run` tag value:

- `tag` — display name (the select option value)
- `key` — slug used inside `importRunId`
- `runOrder` — relative order discovered from the spreadsheet column position

This file is a reference/sanity artifact. The importer does not read it; Notion select options are created on demand as run cards are written.

## Status Derivation

`record.status` is a single-value execution status, set in `prepare-import.mjs`
with this precedence:

- `Skipped` — the run's Person value matches `/skip/i` (e.g. `skip`, `skip (AP)`,
  `SKIPPED`).
- `Done` — otherwise, when the run's `OK?` value is TRUE.
- empty — otherwise (`OK?` was FALSE / N/A / blank); the `Status` property is
  omitted on import.

Skipped takes precedence over Done. There is **no** OK checkbox — pass/"done"
state is folded into `Status`.

## Assignee Normalization

`prepare-import.mjs` (`deriveAssignment()` / `classifyPerson()`) normalizes the
Person column against a hard-coded `CANONICAL_ASSIGNEES` list:

> Andrew, Gordon, Andy, Bharani, Dirk, Hatton, Jeffrey, JohnT, Samuel, Sue,
> Suzanne, Steve, Noel, Marlon, Heather, Colin

Plus an alias map (`ASSIGNEE_ALIASES`): `SteveMc` -> `Steve`.

Each Person value is classified:

- **skip** (matches `/skip/i`) -> `Status = Skipped`; no assignee, no note. Any
  extra text on a skip value is dropped.
- **`a/b`** where every slash-separated part is canonical -> all parts become
  assignees.
- **starts with a canonical name** (on a word boundary) -> that name is the
  assignee. If there is extra text after the name, the **full original value**
  is kept as a note.
- **anything else** -> no assignee; the full value is kept as a note.

Each run card has exactly one execution (one suite-run column), so this applies
to that single Person value. **Assignee is cleared on Skipped runs.**

## Area

The spreadsheet groups tests under section-header rows. `round2/area-mapping.json`
maps each header's text (matched against the row's description column) to a
`kind`:

- `area` — sets the current functional area (e.g. `Authoring`, `Publishing: PDF`).
- `instruction` / `ignore` — recognized but not yet acted on (area-only scope).

`prepare-import.mjs` scans rows in order; each `area` header updates a running
`currentArea` that is stamped onto every test case beneath it (until the next
`area` header), and imported into the `Area` select. Cases before the first
header would have an empty area. Stale mapping keys that match no row are
harmless.

## Case Summaries

The card title is a short, human-written summary, not a mechanical derivation.
`round2/case-summaries.json` is an array of
`{ "snapshot": <caseSnapshot text>, "summary": <≤5–6 word title> }` entries.
`prepare-import.mjs` looks up each case's `caseSnapshot` and uses the match as
`caseSummary` (the card title); snapshots with no entry get a blank title. The
file is authored incrementally — `prepare-summary.json` reports
`summarizedSnapshotCount` / `unsummarizedSnapshotCount` so coverage is visible
(~599 distinct snapshots total).

## Date Handling

A date cell resolves to `testedOn` (a real date) when it parses, including the
year inference below. When it does **not** resolve:

- a bare `ok` (an OK flag that landed in the date cell) is **dropped** entirely —
  no note, no date warning;
- any other leftover text is added to the card's `notes` (joined with `; ` if
  there are multiple notes) and mirrored into `date-warnings.json`.

### Year inference

Some source dates are a bare `M/D` with no year (`normalizeDate` flags these
`missing-year`). `prepare-import.mjs` fills them inline using a **hard-coded**
`SUITE_RUN_YEARS` map (suiteRunKey -> year), determined once from each affected
run's fully-dated entries:

| suiteRunKey | year |
| --- | --- |
| `5-4` | 2022 |
| `5-3` | 2022 |
| `5-5` | 2023 |
| `4-9` | 2020 |

The inferred value is filled silently into `testedOn` (so it flows to `Tested
On`) and the run drops out of `date-warnings.json`. The only trace is
`inferredYearCount` in `prepare-summary.json`. The year is **not** re-derived at
run time — update the map if a new suite run records bare `M/D` dates.

## Live Notion Database

The target database lives under `parentPageId` in `round2/notion-config.json`. On a clean slate, leave `databases.testCaseRuns` empty and the importer creates the `Test Case Runs` database with the schema below, then records its id in `notion-state.json`.

When the database already exists, the importer reconciles its live schema enough to support the current round2 contract. This document describes the live property contract the importer expects today.

### Notion Mapping: `Test Case Runs`

Database title property name:

- `Test Case Run`

Properties written by `buildCaseRunProperties()`:

- `Test Case Run` -> Notion `title`
  From `record.caseSummary`

- `Import Run ID` -> Notion `rich_text`
  From `record.importRunId`

- `Test Case ID` -> Notion `number`
  From `record.testCaseId`

- `Test Suite Run` -> Notion `select`
  From `record.suiteRunTag`. The closed, selectable suite-run list. Option names cannot contain commas (commas are replaced with spaces).

- `Area` -> Notion `select`
  Optional, from `record.area`

- `Legacy Number` -> Notion `rich_text`
  From `record.legacyNumber`

- `Dokimion ID` -> Notion `rich_text`
  From `record.dokimionId`

- `Past Issues` -> Notion `rich_text`
  From `record.pastIssues`, with `BL-1234` references linked to `https://issues.bloomlibrary.org/youtrack/issue/BL-1234`

- `Priority` -> Notion `select`
  Optional, from `record.priority`. Options pre-defined with colors on create: `1` red, `2` orange, `3` yellow, `Ignore` gray, `Duplicate` brown.

- `Est. Time (min)` -> Notion `number`
  Optional, from `record.estTimeMin`

- `Assignee` -> Notion `multi_select`
  From `record.assignees` (canonical tester names; empty on Skipped runs)

- `Notes` -> Notion `rich_text`
  From `record.notes`

- `Build Tested` -> Notion `rich_text`
  From `record.buildTested`

- `Issue Links` -> Notion `rich_text`
  From `record.issueLinks`, with `BL-1234` references linked to `https://issues.bloomlibrary.org/youtrack/issue/BL-1234`

- `Status` -> Notion `select`
  Optional, from `record.status` (`Skipped` / `Done`). Options pre-defined with colors on create: `Done` green, `Skipped` gray.

- `Source Row Number` -> Notion `number`
  From `record.sourceRowNumber`

- `Tested On` -> Notion `date`
  Optional, from `record.testedOn`

Run-card page body behavior:

- a `Test Case Snapshot` heading followed by `record.caseSnapshot` as paragraph blocks (one per line; the execution data lives entirely in the card's properties)
- by default the body is only written when empty; set `ROUND2_REPLACE_BODY=1` to delete and rewrite it

## Live Schema Reconciliation

`import-to-notion.mjs` can reconcile the live `Test Case Runs` schema before import (enabled by default; set `ROUND2_RECONCILE_SCHEMA=0` to skip).

Current behavior:

- drops the obsolete `Test Case` relation if present
- drops `Test Suite Run` if it still exists as a `relation`, then re-adds it as a `select`
- adds any of the required run properties that are missing (see `REQUIRED_RUN_PROPERTIES` in the importer)

The old `Test Cases` and `Test Suite Runs` databases are never modified.

## Import State And Identity

The importer is rerunnable. Run cards are keyed by `importRunId` in `notion-state.json`.

If a page ID is already present in `notion-state.json`, the importer updates that page. If not, and lookup is allowed, it queries the live database by `Import Run ID`.

## Page Body Replacement Behavior

- default mode: only append blocks if the page body is empty
- replacement mode (`ROUND2_REPLACE_BODY=1`): delete existing child blocks and rewrite the body

Use replacement mode when the prepared body format changes and old content should be discarded.

## Useful Environment Variables

### Preparation

- `ROUND2_LIMIT_CASES`
  Limit the number of importable cases processed (default `10`; set `0` for no limit).

### Import

- `ROUND2_ALLOW_LOOKUP`
  Enable live lookup by `Import Run ID` when state is missing (default on; set `0` to disable).

- `ROUND2_RECONCILE_SCHEMA`
  Reconcile the live `Test Case Runs` schema (default on; set `0` to disable).

- `ROUND2_RESET_EXISTING`
  Archive existing pages in the `Test Case Runs` database before import.

- `ROUND2_REPLACE_BODY`
  Replace existing page-body content instead of leaving it alone.

- `ROUND2_RETRY_COUNT`, `ROUND2_RETRY_DELAY_MS`, `ROUND2_REQUEST_TIMEOUT_MS`
  Notion API retry/backoff/timeout tuning.

## Auxiliary Output Files

- `prepare-summary.json`
  Compact summary of the preparation pass: source CSV path, output dir, case limit, slot count, case count, suite-run tag count, run count, date-warning count, inferred-year count, and duplicate-importRunId count (should be 0).

- `date-warnings.json`
  Dates that didn't parse and weren't a bare `ok` (see Date Handling). The same leftover text is also added to the card's `notes`, so this is a diagnostic mirror.

- `notion-state.json`
  Persistent mapping from `importRunId` to Notion page ID. Operational state, not a source document.

- `notion-import-failures.json`
  Operational error log. An empty `caseRuns` array means the latest import pass completed cleanly.

## Recommended Mental Model For Future Work

When changing round2, use this order of concern:

1. Improve preparation quality in `prepare-import.mjs`.
2. Keep `import-to-notion.mjs` focused on mapping and transport into the single database.
3. Treat the `Test Suite Run` select tag as the only suite-run modeling.
4. Verify against the live Notion schema, not just this document.

## Practical Scratch-Start Checklist

If you had no session memory and needed to resume this work from scratch:

- `round2/prepare-import.mjs` creates `test-case-runs.json` plus `suite-run-tags.json`, a summary, and a warnings file
- `round2/import-to-notion.mjs` uploads `test-case-runs.json` into the single `Test Case Runs` database defined in `round2/notion-config.json`
- there is only one Notion database now
- suite-run membership is the `Test Suite Run` select tag, not a relation
- case metadata is folded onto each run card as its own properties
- execution outcome is the `Status` select (`Skipped` / `Done`), derived from skip markers and the spreadsheet `OK?` column; there is no OK checkbox
- the live Notion schema matters more than older design assumptions

That is the round2 contract.
